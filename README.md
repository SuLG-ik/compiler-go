# Лабораторная работа №5. Построение AST и проверка контекстно-зависимых условий

## 1. Название работы и автор

**Тема:** Построение абстрактного синтаксического дерева (AST) и реализация
семантического анализатора для конструкции объявления функции языка Kotlin.

**Автор:** Ненашкин Владимир Денисович, группа АВТ-313, НГТУ, АВТФ.

## 2. Вариант задания

Cоздание функции языка Kotlin.

Анализируемая конструкция:

```kotlin
fun имя_функции(параметры): Тип {
    return выражение
};
```

Допустимые типы параметров и возвращаемого значения: `Int`, `Boolean`,
`Float`, `Double`. В выражении возврата допустимы идентификаторы,
целочисленные и вещественные литералы, операции `+`, `-`, `*`, `/` и
скобочные группы.

### Примеры верных строк

```kotlin
fun calc(a: Int, b: Int, c: Int): Int {
    return a + (b * c)
};
```

```kotlin
fun mix(a: Int, b: Float): Float {
    return a + b
};
```

```kotlin
fun isReady(flag: Boolean): Boolean {
    return flag
};
```

## 3. Контекстно-зависимые условия

Семантический анализатор выполняет четыре группы проверок. Сообщения
формируются с указанием строки, столбца и фрагмента (имя или литерал),
вызвавшего ошибку.

| № | Правило | Пример исходного кода | Ожидаемое сообщение |
|---|---|---|---|
| 1 | **Уникальность имён** — параметры функции должны иметь различные имена в пределах одной области видимости | `fun f(a: Int, a: Int): Int { return a };` | `Идентификатор уже объявлен ранее в списке параметров` (фрагмент: `a`) |
| 2 | **Совместимость типов** — тип возвращаемого выражения должен соответствовать объявленному типу функции (допускается расширение `Int → Float → Double`) | `fun f(a: Float): Int { return a };` | `Несовместимость типов: ожидается Int, получено Float` |
| 3 | **Допустимые значения** — числовые литералы должны входить в допустимый диапазон (`Int`: −2³¹…2³¹−1, `Float`: диапазон IEEE‑754) | `fun f(): Int { return 9999999999999999 };` | `Целочисленный литерал выходит за допустимые пределы типа Int` |
| 4 | **Использование объявленных идентификаторов** — все идентификаторы выражения возврата должны быть объявлены среди параметров | `fun f(a: Int): Int { return a + b };` | `Используется необъявленный идентификатор` (фрагмент: `b`) |

Дополнительная проверка типов в выражении: операнды операций `+`, `-`, `*`,
`/` должны быть числовыми. При попытке использовать в арифметике параметр
типа `Boolean` фиксируется ошибка `Операция '+' не применима к типу Boolean`.

Примеры файлов с семантическими ошибками подготовлены в каталоге
[docs/examples](docs/examples/):

- [semantic_01_duplicate_param.kt](docs/examples/semantic_01_duplicate_param.kt) — повторное объявление параметра;
- [semantic_02_undeclared_ident.kt](docs/examples/semantic_02_undeclared_ident.kt) — необъявленный идентификатор;
- [semantic_03_return_type_mismatch.kt](docs/examples/semantic_03_return_type_mismatch.kt) — несовместимость типов;
- [semantic_04_boolean_in_arith.kt](docs/examples/semantic_04_boolean_in_arith.kt) — `Boolean` в арифметическом выражении;
- [semantic_05_int_overflow.kt](docs/examples/semantic_05_int_overflow.kt) — выход целого литерала за пределы `Int`.

## 4. Структура AST

### 4а. Типы узлов

Дерево строится из обобщённого узла `AstNode`, у которого хранятся: вид
узла (`Kind`), упорядоченный список полей (`Fields`) и позиция в исходном
тексте. Поле может содержать либо скалярный атрибут (`Label: Value`), либо
вложенный узел.

| Узел | Назначение | Атрибуты | Дочерние узлы |
|---|---|---|---|
| `FunctionDeclNode` | объявление функции | `name` | `params: ParamListNode`, `returnType: TypeNode`, `body: ReturnStmtNode` |
| `ParamListNode` | список формальных параметров | — | произвольное число `ParamNode` |
| `ParamNode` | формальный параметр | `name` | `type: TypeNode` |
| `TypeNode` | тип параметра или возвращаемого значения | `name` | — |
| `ReturnStmtNode` | оператор `return` | — | `expr` (выражение) |
| `BinaryOpNode` | бинарная операция `+`, `-`, `*`, `/` | `op` | `left`, `right` |
| `IdentifierNode` | использование идентификатора в выражении | `name` | — |
| `IntLiteralNode` | целочисленный литерал | `value` | — |
| `FloatLiteralNode` | вещественный литерал | `value` | — |

Объявленные параметры регистрируются в таблице символов `SymbolTable`,
которая поддерживает методы `Declare`, `Lookup` и иерархию областей
видимости через ссылку на родительскую таблицу.

### 4б. Графический AST для верной строки

После выполнения семантического анализа пользователь может открыть
отдельное окно **«Показать AST»** из панели инструментов или из меню
**Пуск → Показать AST**. В этом окне AST отображается не как текст, а как
графическое дерево: каждый узел оформлен отдельным блоком, внутри которого
показаны тип узла и ключевые атрибуты, а линии между блоками отражают
отношение «родитель → потомок».

Ниже показан пример графического AST для корректной функции
`fun calc(a: Int, b: Int, c: Int): Int { return a + (b * c) };`:

### 4в. Формат вывода AST в программе

AST выводится в текстовом виде, аналогичном утилите `tree`: безымянные
дочерние элементы отображаются как `├──`/`└──`, скалярные атрибуты — в
виде `label: значение`. Для приведённой выше строки результат выглядит
следующим образом:

```
FunctionDeclNode
├── name: "calc"
├── params: ParamListNode
│   ├── ParamNode
│   │   ├── name: "a"
│   │   └── type: TypeNode
│   │       └── name: "Int"
│   ├── ParamNode
│   │   ├── name: "b"
│   │   └── type: TypeNode
│   │       └── name: "Int"
│   └── ParamNode
│       ├── name: "c"
│       └── type: TypeNode
│           └── name: "Int"
├── returnType: TypeNode
│   └── name: "Int"
└── body: ReturnStmtNode
    └── expr: BinaryOpNode
        ├── op: "+"
        ├── left: IdentifierNode
        │   └── name: "a"
        └── right: BinaryOpNode
            ├── op: "*"
            ├── left: IdentifierNode
            │   └── name: "b"
            └── right: IdentifierNode
                └── name: "c"
```

В графическом интерфейсе AST размещается во вкладке «Результаты» вместе
с заголовком об успешном завершении анализа или о количестве найденных
ошибок. Семантические ошибки выводятся во вкладке «Ошибки» в виде таблицы
со столбцами «Фрагмент», «Местоположение» и «Описание»; в строке итогов
указано общее количество ошибок.

### 4г. Использованные графические средства

Для визуализации AST в приложении применяются следующие средства:

- отдельное модальное окно `AstViewerModal`, открываемое по кнопке
    **«Показать AST»**;
- преобразование текстового AST, построенного бэкендом, во внутреннюю
    древовидную структуру на стороне React;
- HTML-разметка на основе вложенных списков (`ul` / `li`) для описания
    иерархии узлов;
- CSS-псевдоэлементы и линии-соединители для отображения рёбер дерева;
- карточки узлов с цветовым выделением типа узла, имени связи (`params`,
    `body`, `left`, `right` и т.д.) и ключевых атрибутов (`name`, `type`,
    `value`, `op`).

Таким образом, одно и то же AST доступно в двух формах: текстовой
(`tree`-подобный вывод) и графической (модальное окно с визуальным деревом).

### 5б. Пример графического AST при семантической ошибке

В следующем примере AST всё равно строится, хотя в списке параметров есть
повторное имя `a`. Ошибка показывается отдельно во вкладке «Ошибки», а

![Графический AST: повторное объявление параметра](docs/ast_graph_duplicate.svg)

В случае лексических или синтаксических ошибок семантический анализ не
запускается: пользователю показывается соответствующее сообщение, а
таблица ошибок заполняется ошибками предыдущих этапов.

Корректность поведения подтверждается автоматическими тестами в файле
[semantic_test.go](semantic_test.go) (запуск `go test ./...`).

## 6. Инструкция по запуску

### Требования

- Go ≥ 1.21;
- Node.js ≥ 18 и npm;
- Wails CLI v2 (`go install github.com/wailsapp/wails/v2/cmd/wails@latest`);
- ANTLR 4 — только при необходимости перегенерации лексера/парсера ANTLR
  (`tools/generate_antlr.sh`).

### Сборка и запуск приложения

```sh
# установка JS-зависимостей
cd frontend && npm install && cd ..

# режим разработки (горячая перезагрузка)
wails dev

# сборка релизной версии
wails build
# готовый бинарь будет в build/bin/
```

Главное окно содержит редактор кода, панель результатов с вкладками
«Результаты» и «Ошибки», команды **Пуск → Семантический анализ** и
**Пуск → Показать AST**, кнопку Σ для запуска семантического анализа и
отдельную кнопку **«Показать AST»** на панели инструментов. Для ускорения
работы также доступны сочетания `Ctrl+Alt+R` и `Ctrl+Alt+A`.

### Запуск тестов

```sh
go test ./...
```

Тесты проверяют как ранее реализованные лексер и парсер, так и
семантический анализатор: уникальность параметров, недопустимые
идентификаторы, несовместимость типов, диапазоны числовых литералов,
численное расширение `Int → Float`, отсутствие AST при ошибках парсера.
